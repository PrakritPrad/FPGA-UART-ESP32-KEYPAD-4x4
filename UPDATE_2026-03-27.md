# System Update Report (2026-03-27)

Repository: FPGA-UART-ESP32-KEYPAD-4x4  
Branch: main

## 1) ภาพรวมระบบปัจจุบัน
ระบบเป็น Digital Door Lock แบบผสม FPGA + ESP32 โดยแยกหน้าที่ดังนี้:

- FPGA:
  - สแกนคีย์แพด 4x4
  - ประมวลผลสถานะล็อก/ปลดล็อก และเปลี่ยนรหัสผ่าน
  - ควบคุม Relay + Buzzer + LED สถานะ
  - ส่ง event code ทาง UART
- ESP32:
  - รับ event code จาก FPGA ทาง UART
  - แสดงสถานะบน LCD I2C
  - จัดการ WiFi และ Telegram Bot
  - แจ้งเตือนเหตุการณ์ (Unlock/Lock/Lockout) ไปยังผู้ใช้หลายคน

## 2) Data Flow ของระบบ
1. ผู้ใช้กดคีย์บน Keypad 4x4
2. โมดูล scanner/encoder ฝั่ง FPGA แปลงเป็น key_code + key_valid
3. FSM ใน FPGA ตัดสินใจสถานะ (LOCKED/UNLOCKED/LOCKOUT/Change PIN)
4. FPGA ส่งรหัสเหตุการณ์ผ่าน UART (เช่น U, K, L และ digit/control บางรายการ)
5. ESP32 อ่าน UART แล้ว:
   - อัปเดตสถานะ LCD
   - จัดคิวแจ้งเตือน Telegram
6. Network task บน ESP32 ส่งข้อความไป Telegram เมื่อมีเน็ต

## 3) สิ่งที่อัปเดตหลักในรอบนี้

### 3.1 FPGA: เพิ่มไฟแสดงสถานะ 2 สี
- เพิ่มพอร์ต LED ใหม่ที่ top-level:
  - RED_LED
  - GREEN_LED
- map พินใหม่ใน UCF:
  - RED_LED -> P21
  - GREEN_LED -> P16

ผลลัพธ์เชิงพฤติกรรม:
- RED LED ติดเมื่ออยู่สถานะล็อก, lockout, หรือช่วง fail ชั่วคราว
- GREEN LED ติดเมื่อปลดล็อก หรืออยู่ขั้นตอนเปลี่ยนรหัสที่อนุญาต

### 3.2 FPGA: ปรับตรรกะ Buzzer และ fail indication
- ยกเลิก PWM ย่อยของ buzzer และใช้ on/off ตามช่วงเวลา event โดยตรง
- เพิ่ม fail_led_timer (ประมาณ 500 ms) เพื่อให้มี feedback ชัดเมื่อใส่รหัสผิด
- จัดรูปแบบเสียงหลัก:
  - สั้น: event ทั่วไปบางกรณี
  - คู่: fail
  - lockout ยังมีพฤติกรรมเตือนชัดตาม state

### 3.3 FPGA: ปรับ event ระหว่าง state transitions
- เมื่อ unlock สำเร็จ ส่ง U
- เมื่อสั่งล็อกกลับ ส่ง K พร้อม beep สั้น
- เมื่อ fail ซ้ำจนครบเงื่อนไข lockout ส่ง L และเข้า timer lockout
- ปรับบางช่วง transition ให้ feedback ทันทีขึ้น

### 3.4 ESP32: เพิ่มความสามารถ Network/Telegram แบบ production มากขึ้น
- เพิ่มไลบรารีและโมดูล:
  - WiFi, WiFiClientSecure
  - UniversalTelegramBot, ArduinoJson
  - SPIFFS
  - FreeRTOS task + queue
- เพิ่มรองรับผู้ใช้หลาย chat id (สูงสุด 10)
- เก็บ chat id ลง SPIFFS (/chats.csv) เพื่อคงข้อมูลข้ามการรีบูต
- เพิ่ม flow ยืนยันผู้ใช้ใหม่ด้วย MASTER_CODE ก่อน subscribe
- ทำ timestamp โดย sync เวลา (NTP, UTC+7)

### 3.5 ESP32: แยกงานเป็น 2 Task
- uartTask (core 1, priority สูงกว่า):
  - อ่าน UART ตลอด
  - อัปเดตสถานะและ countdown lockout
  - render LCD เฉพาะตอนข้อมูลเปลี่ยน (ลดกระพริบ)
- networkTask (core 0):
  - maintain WiFi แบบ non-blocking
  - poll Telegram ตามช่วงเวลา
  - ดึง alert จาก queue แล้วส่งออกตาม interval

ข้อดี:
- การอ่าน UART ไม่ถูกบล็อกด้วยงานเน็ต
- แจ้งเตือนมีคิว รองรับสภาวะเน็ตไม่นิ่งได้ดีขึ้น
- UX บน LCD ลื่นขึ้นเพราะไม่ clear หน้าจอทุกครั้ง

## 4) Event Mapping ที่ระบบใช้ตอนนี้
- จาก FPGA/Lock FSM (ตัวอย่างสำคัญ):
  - U = Unlock success
  - K = Lock command/return to locked
  - L = Lockout entered
  - O = Wrong password
  - G = Enter old-PIN verification flow
  - N = Enter new-PIN set flow
  - A = Backspace
  - C = Clear
  - 0-9 = Digit input echo

- การแปลผลใน ESP32:
  - U -> แสดง UNLOCK + ส่ง Telegram "ตู้เซฟถูกเปิด"
  - K -> แสดง LOCK + ส่ง Telegram "ตู้เซฟถูกปิดแล้ว"
  - L -> แสดง LOCKOUT + ส่ง Telegram เตือนความเสี่ยง

## 5) สถานะที่ผู้ใช้งานจะเห็น
- หน้าจอ LCD บรรทัด 1: ST:<state>
- หน้าจอ LCD บรรทัด 2:
  - โหมดทั่วไป: PIN:____
  - โหมด lockout: WAIT:xxs
- Relay:
  - ปลดล็อก = เปิดวงจรตาม active logic ใน top/fsm
- LED:
  - แดง = lock/lockout/fail indication
  - เขียว = unlock/ช่วงอนุญาตเปลี่ยนรหัส
- Telegram:
  - แจ้งเตือน unlock, lock, lockout พร้อมเวลา

## 7) ข้อสังเกตเชิงความปลอดภัย (สำคัญ)
ใน esp32/esp32.ino ปัจจุบันมีการเก็บข้อมูล credential แบบ hardcode:
- WiFi SSID/PASSWORD
- Telegram bot token
- MASTER_CODE

คำแนะนำ:
- ย้ายค่าเหล่านี้ไปไฟล์ config ที่ไม่ commit (หรือใช้ provisioning)
- rotate bot token หลังทดสอบเสร็จ
- เพิ่ม allowlist/ACL สำหรับ user ที่ลงทะเบียน

## 8) สรุปสั้น
เวอร์ชันนี้ยกระดับจาก keypad lock ทั่วไป ไปเป็นระบบควบคุมที่มี
- feedback ทางกายภาพครบ (Relay/Buzzer/LED)
- feedback ทางหน้าจอ (LCD)
- feedback ทางเครือข่าย (Telegram)
พร้อมโครงสร้าง task แยกที่ทำให้ระบบตอบสนองดีขึ้นและพร้อมต่อยอดใช้งานจริงมากขึ้น.
